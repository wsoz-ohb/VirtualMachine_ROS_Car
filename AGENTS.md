# Repository Guidelines

## Project Structure & Module Organization
- `main/main.c` boots the firmware, initializes FreeRTOS tasks, and delegates motor control to the app layer.
- `main/myapp/` contains the high-level motion orchestration in `motor_app_.c` and its public interface `motor_app.h`.
- `components/` houses reusable drivers: `tb6612_motor` for the dual H-bridge, `my_pid` for controller logic (`inc/` headers, `src/` sources), and `my_encoder` for wheel feedback.
- `build/` and `sdkconfig` are generated by ESP-IDF; keep them out of manual edits except through `idf.py menuconfig`.

## Build, Test & Development Commands
- `idf.py set-target esp32s3` — aligns the toolchain with the configured target (run once per environment).
- `idf.py build` — configures CMake and compiles all components.
- `idf.py -p COMx flash monitor` — flashes firmware to the board and tails the UART log; replace `COMx` with your port.
- `idf.py fullclean` — wipes the build cache when switching SDK versions or resolving corrupted artifacts.
- `idf.py menuconfig` — adjust PWM pins, logging verbosity, and other runtime options before rebuilding.

## Coding Style & Naming Conventions
- Follow ESP-IDF C conventions: 4-space indentation, braces on new lines for functions, and `snake_case` for functions and static symbols.
- Keep public headers under `components/<name>/inc` with include guards (`MY_PID_H` style); place implementation in matching `src/`.
- Use `ESP_LOGx(TAG, ...)` macros with uppercase tags declared near the top of each file.
- Prefer descriptive GPIO constants (`TB6612_IN1_GPIO`) and document hardware assumptions inline.

## Testing Guidelines
- No automated tests ship today; add Unity-based tests under `components/<name>/test/` and register them with ESP-IDF’s unit test target.
- Mirror source filenames when naming tests (e.g., `test_tb6612_motor.c`) and ensure they exercise boundary PWM values and PID saturation limits.
- Before opening a PR, run `idf.py build` and validate motor behavior on hardware; capture monitor logs showing initialization and movement.

## Commit & Pull Request Guidelines
- Git metadata is not bundled here; adopt Conventional Commits (`feat: add encoder calibration`, `fix: guard queue overflow`) for clarity.
- Squash work-in-progress commits locally and ensure diffs touch only relevant modules.
- Pull requests should summarize the change, list tested hardware (board revision, power setup), link any tracking tickets, and attach `idf.py build` output or key monitor excerpts.

## Configuration & Security Notes
- Store sensitive credentials (Wi-Fi, tokens) in a local `sdkconfig.defaults` override and exclude them from version control.
- When changing `sdkconfig`, note the impact on pin mappings and keep the intended board revision documented in the PR.
